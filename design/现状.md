结论
没有“都实现”。当前 openfang/crates 已经把 Parlant 思想里的控制层骨架接进了主对话链路，但它还是一个可运行的 MVP/近似实现，不是对 F:\AIWorkspace\parlant\src\parlant 语义控制层的等价 Rust 迁移。

关键缺口

guideline_relationships 这一层现在有明显断层。控制面 API 注释/入参用的是 overrides | complements | conflicts_with | requires，但运行时 resolver 真正识别的是 depends_on | excludes | prioritizes_over，这意味着按当前 API 语义创建出来的关系大概率不会参与生效计算。control_routes.rs lib.rs
工具授权模型没有保持 Parlant 最核心的“guideline -> tool association”约束。Parlant 里工具是 guideline 关联的一等对象，且“只有命中 guideline 才该暴露工具”。guideline_tool_associations.py tools.md 现在 openfang 用的是 tool_exposure_policies，并且 ToolGate 对“无策略工具”明确是 open by default，这和 Parlant 语义不等价。lib.rs lib.rs
Parlant 的“工具执行后触发 reevaluation / additional preparation iteration”这一条主链路现在已经基本接上：默认非 streaming LLM 路径和 `send_message_streaming` 都会先走 `compile_turn_iterative`，并在 tool result 后通过 iterative callback 把 `prior_tool_calls` 回灌给 control-plane 做下一轮 guideline / tool visibility 重编译。当前更大的剩余缺口已经转到动态 context variable 与 retriever binding 的控制面写入口，而不是 preparation iteration 本身。engine.py entity_cq.py lib.rs kernel.rs
Context variable 只实现了静态子集。openfang 里非 static 变量会直接变成 "<unresolved:...>" 占位字符串；而 Parlant 的变量模型有 tool_id、freshness_rules 和 value 级别的持久化 CRUD。store.rs context_variables.py
Glossary / retriever 也还是简化版。openfang 会把 scope 下所有 glossary term 全量加载，不做 Parlant 那种 relevant-term 选择；retriever runtime 会读取 retriever_bindings，但控制面 API 只暴露了 retriever 的 create/list，没有 binding 的写入口，所以按 guideline/journey 绑定 retriever 这件事实际上不可运营。store.rs glossary.py store.rs control_routes.rs
Journey 和 guideline 数据模型都有明显缩水。Parlant 的 journey 是 graph，有 node-level tools、conditions、composition_mode、priority 等 richer semantics；guideline 还有 metadata、criticality、labels、track。journeys.py guidelines.py openfang 现在的 GuidelineDefinition / JourneyDefinition 更像轻量规则 + SOP 状态机，journey state 只是投射 guideline_actions 文本，不是完整 journey graph 语义。control.rs store.rs
已实现部分

这套控制层不是摆设，已经接进主链路了。server 会构建 coordinator 并挂到 kernel；kernel 在默认非 streaming 与 streaming 两条 LLM 链路上都会先走 `compile_turn_iterative`，按结果裁剪 tool 集、注入 journey/guideline/knowledge/context variable 到 prompt，并在 tool result 后继续通过 iterative callback 重编译，再在轮后写 explainability / journey transition 记录。server.rs kernel.rs
